# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Язык общения:** Всегда отвечай на русском языке.

## Обзор системы

Мультиагентная система аудита проектной документации жилых зданий (МКД). Центральный оркестратор (Claude Code) запускает параллельных специализированных агентов через `Agent` tool. Каждый агент анализирует проект со своей экспертной позиции, затем агенты-критики перекрёстно проверяют находки, и синтезатор формирует финальный отчёт.

**Принцип:** один MD-файл → N параллельных агентов → спор → синтез → результат.

## Правило синхронизации промптов (RU ↔ EN)

Промпты хранятся в двух зеркальных папках:
- **`Prompts_EN/`** — **рабочая версия**, используется агентами при аудите
- **`Prompts_RU/`** — русский перевод для чтения и редактирования пользователем

**Имена файлов идентичны** в обеих папках (без суффиксов `_RU`/`_EN`).

**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО:** если пользователь изменил файл в `Prompts_RU/`, оркестратор **ОБЯЗАН напомнить** пользователю, что нужно применить аналогичное изменение в `Prompts_EN/` — и предложить автоматически перенести изменения. Английская версия — это та, которую читают агенты. Рассинхронизация RU и EN приведёт к тому, что агенты будут работать по устаревшим инструкциям.

**При обнаружении изменений в RU-файле:**
1. Сообщить: «Файл `Prompts_RU/{SECTION}/{file}.md` изменён. Нужно обновить EN-версию `Prompts_EN/{SECTION}/{file}.md`.»
2. Предложить: «Перенести изменения автоматически?»
3. Если да — применить аналогичные правки в EN-файле

## Команды

```bash
# Генерация Excel-отчёта
python3 "Audits/gen_report.py" "projects/EOM/133_23-ГК-ГРЩ"
```

Основной запуск аудита — через Claude Code CLI: пользователь указывает путь к проекту, оркестратор читает `project_info.json` и запускает пайплайн.

### Формат project_info.json

```json
{
  "project_id": "133_23-ГК-ЭМ2",
  "section": "EOM",
  "md_files": ["Часть 1..._document.md", "Часть 2..._document.md"]
}
```

- `section` — **обязан** совпадать с именем папки в `Prompts_EN/` (EOM, AI, AR, VK, OV, TX, GP, SS, PT, ITP, KM, POS)
- `md_file` (строка) — один документ, обособленный проект
- `md_files` (массив) — многочастный проект, части идут по порядку (Часть 2 — продолжение Части 1)

**Правило для оркестратора:** если в project_info есть `md_files` — конкатенировать все MD-файлы в один поток перед Раундом 0. Нумерация страниц второго файла продолжает первый. Если есть только `md_file` — один документ.

### Раунд 0: Подготовка и Vision

#### Шаг 0а: Чтение входных данных

1. Читает `project_info.json` → определяет `section` и список MD-файлов
2. Если `md_files` (массив) — конкатенирует все части в один поток (Часть 2 продолжает Часть 1)
3. Если `md_file` (строка) — один документ

#### Шаг 0б: Скачивание IMAGE-блоков

Если папка `_output/blocks/` пуста или отсутствует, оркестратор скачивает кропы изображений:

1. Находит HTML-файл в папке проекта (`*_ocr.html`)
2. Парсит все ссылки вида:
   ```
   href="https://pub-...r2.dev/tree_docs/{DOC_UUID}/crops/{BLOCK_ID}.pdf"
   ```
3. Скачивает каждый PDF-кроп в `_output/blocks/{BLOCK_ID}.pdf`
4. Проверяет: количество скачанных файлов = количество `BLOCK [IMAGE]` в document.md

**Если `_output/blocks/` уже содержит файлы** — пропускает скачивание (блоки уже готовы).

#### Шаг 0в: Vision (Gemini)

На данный момент Gemini-агенты запускаются **отдельно** от оркестратора. В будущем — интеграция в пайплайн.

1. Каждый IMAGE-блок получает свой vision-агент (Gemini)
2. Агент получает кроп из `_output/blocks/{BLOCK_ID}.pdf` + промпт `R0_vision_{SECTION}.md`
3. Результаты собираются в `_output/structured_blocks.json`

**Если `_output/structured_blocks.json` уже существует** — оркестратор пропускает Vision и переходит к шагу 0г.

#### Шаг 0г: Обогащение document.md

Оркестратор создаёт `_output/document_enriched.md` — копию document.md, в которой каждый блок `### BLOCK [IMAGE]: {ID}` заменён на структурированное описание из `structured_blocks.json`.

**Было** (OCR-описание, нечитаемое для анализа):
```markdown
### BLOCK [IMAGE]: 4W69-GN7E-MQ6
**[ИЗОБРАЖЕНИЕ]** | Тип: Схема
**Краткое описание:** Схема расчетная однолинейная ВРУ-НС...
**Описание:** Чертеж представляет собой развернутую однолинейную...
**Сущности:** ВРУ-НС, НС.РП1, НС.РП2...
```

**Стало** (структурированные данные от Vision):
```markdown
### BLOCK [IMAGE]: 4W69-GN7E-MQ6
**Тип:** schema | **Страница:** 10

ИСТОЧНИК:
- Ввод питания от ГРЩ, кабель М-1.6 ППГнг(А)-HF 5х4 L=15м.

ВВОД:
- Щит: ЩСН ЩРН-18-IP31 350x300x120.
- Вводной аппарат: выключатель нагрузки BP, C 40 A.

ОТХОДЯЩИЕ ЛИНИИ:
Линия 1 (Гр.1):
  - Автомат: QF1, BA-103 1P, C 10 A
  - Кабель: ППГнг(А)-HF 3x1.5
  ...
```

**Правила замены:**
1. Найти каждый `### BLOCK [IMAGE]: {ID}` в document.md
2. Найти этот ID в structured_blocks.json
3. Заменить всё содержимое блока (от `### BLOCK [IMAGE]` до следующего `### BLOCK` или `## СТРАНИЦА`) на описание из structured_blocks
4. Если ID не найден в structured_blocks.json → оставить оригинальный OCR-текст, добавить пометку `⚠️ Vision-описание отсутствует`
5. Оригинальный document.md **не изменяется** — enriched создаётся отдельно

**Если `_output/document_enriched.md` уже существует** — пропускает обогащение.

Все последующие шаги (парсинг листов, нарезка, передача агентам) работают с `document_enriched.md`, а не с оригинальным `document.md`.

#### Шаг 0д: Парсинг листов

Оркестратор читает `document_enriched.md` и строит индекс листов. Маркер листа:

```
## СТРАНИЦА {N}
**Лист:** {номер}
**Наименование листа:** {название}
```

Для каждого листа фиксируется: номер страницы, номер листа, наименование, диапазон строк (от `## СТРАНИЦА N` до следующего `## СТРАНИЦА`).

#### Шаг 0е: Классификация листов

#### Шаг 0ж: Маппинг категорий → агентов

Для каждого R1-агента определено, какие категории листов ему нужны. **Общие листы** (`general_notes`, `specification`) дублируются всем агентам — это ожидаемо и правильно.

| `fire_safety` | `schema`, `specification`, `general_notes`, `fire_safety` | `lighting_plan`, `cable_route_plan` |
| `consistency` | `toc`, `schema`, `specification`, `general_notes` | все остальные (получает ВЕСЬ document.md) |
| `tables` | `load_table`, `schema`, `specification`, `general_notes` | `cable_journal` |
| `norms` | все (получает ВЕСЬ document.md) | — |
| `lighting` | `lighting_plan`, `schema`, `specification`, `general_notes` | `facade_plan` |
| `cable_routes` | `cable_route_plan`, `specification`, `general_notes` | `power_plan`, `lighting_plan` |
| `automation` | `schema`, `specification`, `general_notes`, `automation` | `lighting_plan` |
| `grounding` | `grounding_plan`, `schema`, `specification`, `general_notes` | `section_drawing` |

**Агенты-исключения** (получают ВЕСЬ document.md):
- `norms` — ищет ссылки на нормы по всему документу
- `consistency` — сверяет данные между источниками (текст, схема, план, спецификация)

Для крупных документов (>500 KB) эти агенты могут запускаться на модели с большим контекстом.

#### Шаг 0з: Сборка срезов и запись файлов (ОБЯЗАТЕЛЬНЫЙ)

Оркестратор **записывает файлы срезов** в `_output/slices/` через Write tool. Каждый R1-агент получит путь к своему персональному срезу, а не к полному документу.

**Если `_output/slices/slice_manifest.json` уже существует** — пропускает нарезку.

##### Алгоритм

1. **Уточнение наименований через ВРЧ.** На шаге 0д оркестратор уже распарсил страницы. Теперь он ищет в листах с категорией `general_notes` или `toc` таблицу «Ведомость рабочих чертежей» (маркер: таблица с колонками «Лист» и «Наименование»). Строит маппинг `номер_листа → описательное_наименование`. Это нужно потому, что OCR-штамп часто содержит только название проекта (например, «Главный распределительный щит»), а ВРЧ — полное наименование листа (например, «Однолинейная расчетная схема ГРЩ»). При классификации (шаг 0е) оркестратор использует наименование из ВРЧ как приоритетное, а OCR-штамп как fallback.

2. **Для каждого НЕ пропущенного R1-агента** (по результатам шага 0к):
   - Если агент имеет `full_document: true` (`norms`, `consistency`) → срез = полный `document_enriched.md`
   - Иначе → собрать страницы, категории которых пересекаются с `required` ∪ `optional` этого агента (по таблице маппинга шага 0ж)
   - Отсортировать собранные страницы по номеру (сохранить оригинальный порядок)

3. **Записать файл среза** `_output/slices/slice_{agent_name}.md` через Write tool:

# Срез для агента: {agent_name}
# Проект: {project_id} | Дисциплина: {section}

{Заголовок документа — всё до первого ## СТРАНИЦА: название, штамп, дата генерации}

**Лист:** {номер}
**Наименование листа:** {наименование}
{содержимое страницы}

```

```

##### Правила сборки срезов

- **Заголовок документа** (штамп, дата) — включается в КАЖДЫЙ срез для контекста
- **Маркеры `## СТРАНИЦА N`** — сохраняются без изменений (агент видит оригинальные номера страниц)
- **Многострочные наименования** — наименование листа может занимать несколько строк (от `**Наименование листа:**` до пустой строки или `### BLOCK`). Оркестратор собирает их в одну строку при записи в `page_index`
- **Страницы без метаданных** (титульные, обложки — без `**Лист:**`) → категория `toc`, включаются только в `consistency` и `norms`
- **Страницы с листом, но без наименования** → наследуют категорию предыдущей страницы (это продолжения листов, например страницы спецификации)

#### Шаг 0з-б: Формирование промпта агента (INLINE-режим)

Оркестратор **сам читает** все файлы агента и **склеивает их содержимое в одну строку промпта**. Агент получает всё в первом сообщении и НЕ делает Read-вызовов — сразу анализирует и пишет результат.

**Почему так:** каждый Read-вызов агента создаёт новый turn API, на котором модель перечитывает ВЕСЬ предыдущий контекст. При 4 Read + анализ + Write = 7 turns, документ перечитывается 6 раз. Inline-режим сокращает до 2 turns (анализ + Write), экономя 50-65% токенов.

**Условие:** inline-режим используется, если суммарный размер всех файлов агента ≤ 400 KB (~100K токенов). Если больше — агент получает пути к файлам и читает их сам через Read tool.

**Порядок блоков в промпте (важен для внимания модели):**

**Для агента `norms`:**
```
=== NORMS DATABASE ===
{содержимое norms/norms_db.json}

=== NORMS PARAGRAPHS ===
{содержимое norms/norms_paragraphs.json}

=== DOCUMENT ===
{содержимое _output/slices/slice_norms.md — полный документ}

=== BASE RULES ===
{содержимое Prompts_EN/common/00_base.md}

=== OWNERSHIP MATRIX ===
{содержимое Prompts_EN/{SECTION}/OWNERSHIP_MATRIX_{SECTION}.md}

=== YOUR TASK ===
{содержимое Prompts_EN/{SECTION}/R1_norms.md}

Write your result to _output/partial_norms.json using the Write tool.
Do NOT read any files — all data is above.
```

**Для всех остальных R1-агентов:**
```
=== BASE RULES ===
{содержимое Prompts_EN/common/00_base.md}

=== OWNERSHIP MATRIX ===
{содержимое Prompts_EN/{SECTION}/OWNERSHIP_MATRIX_{SECTION}.md}

=== DOCUMENT ===
{содержимое _output/slices/slice_{agent_name}.md — персональный срез}

=== YOUR TASK ===
{содержимое Prompts_EN/{SECTION}/R1_{agent_name}.md}

Write your result to _output/partial_{agent_name}.json using the Write tool.
Do NOT read any files — all data is above.
```

**Правила порядка:**
- Правила и матрица — в начале (модель хорошо помнит начало)
- Документ — в середине (данные для поиска)
- Промпт агента — в конце (модель следует последним инструкциям точнее)
- Для `norms`: norms_db идёт первым (самый большой файл, нужен как справочник)

**ВАЖНО:**
- `norms_db.json` и `norms_paragraphs.json` — **только** агенту `norms`
- Агенту в промпте написано "Do NOT read any files" — чтобы он не делал лишних Read
- Единственный tool-вызов агента — Write для записи partial_*.json

#### Ограничение tool calls

В конце каждого inline-промпта добавлять:
```
STRICT BUDGET: You have exactly 1 tool call — Write to save your result.
Do NOT make Read calls, do NOT split analysis across messages.
Analyze everything above, then Write the JSON result in a single action.
```

Для агента `norms` (Read-режим) ограничение мягче — он должен прочитать файлы, но не больше 10 tool calls.

#### Шаг 0и: Проверка покрытия (серые зоны)

После маппинга оркестратор проверяет:

```
Для каждого листа в индексе:
  если лист (по его категориям) не попадает ни в один required/optional ни одного агента:
    → WARNING: "Лист {N} '{наименование}' — серая зона, не покрыт ни одним агентом"
    → Оркестратор назначает лист ближайшему по тематике агенту в его optional
```

Ни один лист не должен остаться без проверки. `uncategorized` листы назначаются агенту `consistency` (он проверяет комплектность).

#### Шаг 0к: Фильтрация агентов (ОБЯЗАТЕЛЬНЫЙ)

После маппинга оркестратор определяет, каких агентов **НЕ нужно запускать**:

```
Для каждого R1-агента:
  если ни одна из его required-категорий НЕ представлена в индексе листов:
    → SKIP: агент не запускается
    → Логировать: "Агент {name} пропущен — нет required-листов ({categories})"
```

**Правило:** агент запускается **только** если хотя бы одна его `required`-категория присутствует в проекте. Отсутствие `optional`-категорий — не повод пропускать агента.

**Исключения** (запускаются всегда):
- `consistency` — проверяет комплектность и расхождения между любыми источниками
- `norms` — проверяет все нормативные ссылки в документе

**Пример для проекта ГРЩ:**
- Нет `lighting_plan` → `lighting` не запускается
- Нет `automation` → `automation` не запускается
- Нет `outdoor` → `outdoor_install` не запускается
- Нет `grounding_plan` → `grounding` не запускается (молниезащита в отдельном томе)
- Есть `schema`, `specification`, `load_table`, `cable_route_plan`, `power_plan` → `cables`, `tables`, `fire_safety`, `cable_routes`, `metering`, `power_equipment` запускаются

Это экономит ~30-40% токенов на проектах, где часть дисциплин вынесена в отдельные тома.

### Раунд 1: Анализ (параллельно, только отфильтрованные агенты)

#### Правило выбора модели для R1-агентов

Оркестратор оценивает размер контекста каждого агента (срез + промпты) и выбирает модель:

| Контекст агента | Модель | Обоснование |
|----------------|--------|-------------|
| ≤ 100K токенов (≤50% контекста Sonnet) | **Sonnet** | Дешевле, быстрее, качество достаточно |
| > 100K токенов | **Opus** | Нужен больший контекст и глубина анализа |

**Оценка размера:** ~4 символа ≈ 1 токен (грубая оценка).
- Срез document_enriched.md: размер файла в байтах / 4
- Промпты (00_base + OWNERSHIP_MATRIX + R1_agent): суммарный размер / 4
- Для агента `norms`: + norms_db.json (~120K токенов) → почти всегда Opus

**Фиксированные модели (не зависят от размера контекста):**
- **Критик (R2)** — всегда **Opus 1M** (большой контекст: все partial_*.json + документ)
- **Синтезатор (R3)** — всегда **Sonnet** (контекст всегда < 100K)
- **Оптимизатор (R4)** — всегда **Opus** (требует инженерной креативности)
- **Критик оптимизации (R4b)** — всегда **Sonnet**

Отфильтрованные R1-агенты запускаются **одновременно**. Агенты полностью независимы друг от друга.

Каждый агент получает **все данные в промпте** (inline-режим, см. шаг 0з-б):
- Правила (00_base + OWNERSHIP_MATRIX) — в начале промпта
- Персональный срез документа — в середине
- Промпт агента (R1_{agent}.md) — в конце
- Агент `norms` дополнительно получает norms_db.json и norms_paragraphs.json

**Агенты НЕ делают Read-вызовов** — все данные уже в контексте. Единственный tool-вызов — Write для записи `_output/partial_<name>.json`.

**Если промпт > 400 KB** (крупный проект + norms_db): агент получает пути к файлам и читает их сам через Read tool (fallback-режим).

#### Обратная связь: missing_data

Если агент не нашёл в предоставленном срезе данные, необходимые для выполнения шага, он сообщает об этом в чеклисте:

```json
"checklist": {
  "step_1_data_collected": {
    "done": true,
    "notes": "...",
    "missing_data": [
      {
        "description": "Кабельный журнал не найден в предоставленных листах",
        "expected_content": "Таблица с перечнем кабелей, длинами и способами прокладки",
        "impact": "Невозможно проверить длины кабелей",
        "steps_affected": ["step_5_installation"]
      }
    ]
  }
}
```

**Логика перезапуска оркестратором:**

1. После завершения всех R1-агентов оркестратор проверяет `missing_data` в каждом `partial_*.json`
2. Если агент запросил данные, и соответствующий лист **существует** в индексе, но не был передан → оркестратор **перезапускает** агента с расширенным срезом
3. Если лист **не существует** в документе → это валидная информация (неполнота комплекта), агент фиксирует это в notes
4. **Максимум 1 перезапуск** на агента — если после второй попытки снова missing_data, оркестратор логирует и продолжает

### Раунд 2: Критика (1 критик Opus 1M)

Один критик на Opus 1M проверяет **все** partial_*.json одновременно. Это позволяет:
- Видеть полную картину — ловить дубли и противоречия между любыми агентами
- Не дублировать partial_consistency.json между критиками
- Упростить пайплайн (один агент вместо трёх)

**Перед запуском критика** оркестратор генерирует norms_subset:
```bash
python3 Audits/build_norms_subset.py "projects/{SECTION}/{PROJECT_ID}"
```

Критик получает **в промпте**:
- Все `_output/partial_*.json`
- `document_enriched.md` (полный — для сверки evidence с любых страниц)
- `_output/norms_subset.json` (только упомянутые нормы, ~6K вместо 120K токенов)
- `Prompts_EN/common/R2_critic.md`

Результат: `_output/review.json`

Вердикты критика (12 типов):

**Принимает:**
- `pass` — замечание обосновано, релевантно, в зоне агента
- `pass_weak_norm` — проблема реальна, но norm_ref слабая → синтезатор скорректирует

**Отклоняет по evidence:**
- `no_evidence` — нет привязки к документу
- `phantom_block` — block_id не существует
- `weak_evidence` — факты не подтверждают вывод
- `contradicts_drawing` — чертёж опровергает замечание
- `arithmetic_error` — ошибка в расчёте агента

**Отклоняет по дублям/конфликтам:**
- `duplicate` — то же самое нашёл другой агент
- `wrong_norm_and_substance` — и норма неверна, и суть ошибочна

**Фильтр релевантности (шаг 6):**
- `reject_irrelevant` — технически верно, но не влияет на деньги/безопасность/эксплуатацию
- `reject_speculative` — предположение с низкой уверенностью, нет убедительных фактов
- `reject_wrong_zone` — замечание вне зоны ответственности агента

### Раунд 3: Синтез (Python-скрипт, без LLM)

Синтез выполняется **кодом**, а не LLM-агентом. Это экономит ~120K токенов.

```bash
python3 Audits/synthesize.py "projects/{SECTION}/{PROJECT_ID}"
```

Скрипт автоматически:
1. Читает `review.json` → фильтрует pass + pass_weak_norm
2. Собирает полные finding-объекты из `partial_*.json`
3. Дедупликация по page + title
4. Пересчёт confidence по правилам (pass +0.1, pass_weak_norm -0.1, multi-agent +0.1)
5. Обогащение norm_quote из `norms/norms_paragraphs.json`
6. Сортировка (Критическое → Экономическое → Эксплуатационное) → нумерация F-001..F-NNN
7. Записывает `_output/findings.json` + `_output/rejected.json`

### Раунд 4: Оптимизация (1 агент Opus + критик Sonnet)

Инженерная оптимизация — не просто замена материалов, а переосмысление проектных решений для удешевления строительства.

Оптимизатор получает **в промпте**:
- `_output/findings.json` — чтобы не оптимизировать ошибочные позиции
- `document_enriched.md` — полный документ (нужна спецификация, чертежи, узлы)
- `norms/vendor_list.json`
- `Prompts_EN/common/R4_optimizer.md`

**Модель:** всегда **Opus** — задача требует инженерного мышления и креативности (см. правило выбора модели в Раунде 1).

**7 категорий оптимизации:**

| # | Категория | Пример |
|---|----------|--------|
| 1 | Замена материалов | Аналог из вендор-листа с теми же параметрами |
| 2 | Оптимизация монтажа | Два лотка на одном кронштейне вместо двух отдельных |
| 3 | Упрощение конструкций | FireBox без лотка вместо огнезащитный короб + лоток |
| 4 | Схемные решения | Объединение двух полупустых щитов в один |
| 5 | Оптимизация трасс | Короче маршрут → меньше кабеля → можно снизить сечение |
| 6 | Технологические альтернативы | Шинопровод вместо пучка параллельных кабелей |
| 7 | Нагрузки и запасы | Завышенное сечение: Iном автомата < 50% Iдоп кабеля |

→ `optimization.json` → критик (Sonnet) → `optimization_review.json`

### Раунд 5: Отчёт

Оркестратор запускает `gen_report.py` с путём к проекту:

```bash
python3 "Audits/gen_report.py" "projects/{SECTION}/{PROJECT_ID}"
```

Скрипт читает `project_info.json`, `findings.json`, `optimization.json`, `optimization_review.json` → генерирует Excel с 3 листами:
- **СВОДКА** — количество замечаний по категориям
- **{project_id}** — все замечания с описанием, решением, категорией
- **ОПТ {project_id}** — предложения оптимизации с вердиктом критика

## Принципы работы агентов

### Единые источники данных

| Тип данных | Источник | Кто использует |
|-----------|---------|---------------|
| Текст + структура чертежей (единый файл) | `_output/document_enriched.md` → нарезается в `_output/slices/slice_{agent}.md` | Агенты R1 (персональный срез), критики (полный документ) |
| Структура схем (оригинал, до обогащения) | `_output/structured_blocks.json` | Оркестратор (для шага 0г), критики (для верификации block_id) |
| Статус нормативных документов | `norms/norms_db.json` | **только** агент `norms` и критик (R2) |
| Проверенные цитаты пунктов | `norms/norms_paragraphs.json` | **только** агент `norms` и синтезатор (R3) |
| Допустимые производители | `norms/vendor_list.json` | optimizer, optimizer_critic |

### Главный принцип: аудитор, а не судья

Агенты выявляют потенциальные проблемы и указывают степень уверенности (`confidence`), а не выносят окончательные вердикты. "Критическое" — только при явном, неоспоримом несоответствии.

### Общие правила (из `Prompts_EN/common/00_base.md`)

1. Читай свой срез `_output/slices/slice_{agent_name}.md` — персональный источник (текст + описания чертежей, только релевантные листы)
2. Каждое замечание ОБЯЗАНО иметь `evidence[]` с привязкой к page/block_id
3. Каждое замечание ОБЯЗАНО иметь `confidence` (0.0-1.0)
4. Если не уверен в норме — ставь `norm_confidence < 0.8`
5. НЕ дублируй чужую работу — ищи только то, что указано в твоём промпте
6. Пиши результат в `_output/partial_<name>.json` с обязательным `checklist`
7. Максимум 20 замечаний от одного агента (приоритизируй)

### Категории замечаний

| Категория | Описание | Приоритет |
|-----------|----------|-----------|
| Критическое | Явное нарушение норм, влияющее на безопасность | 1 (высший) |
| Экономическое | Влияет на стоимость: пересортица, ошибки при закупке/монтаже | 2 |
| Эксплуатационное | Влияет на работу объекта: проблемы при эксплуатации | 3 |

### Нормативная база

1. НЕ ссылайся на устаревшие нормы без пометки о статусе
2. НЕ придумывай номера пунктов — ставь `norm_confidence: 0.5` если не уверен
3. Приоритет: ФЗ → Техрегламенты → СП обязательные → СП добровольные → ГОСТ → ПУЭ
4. При ссылке на ПУЭ-7 — подкрепляй параллельной ссылкой на действующий СП/ГОСТ
5. Перед ссылкой — ставь `norm_confidence` по своей уверенности (агент `norms` проверит статус)

## Механизм спора (debate protocol)

Агенты не общаются напрямую. Спор реализован через файлы:

```
1. Агент-кабели: "Сечение 2.5 мм², нужно 4 мм²" → partial_cables.json
2. Агент-чертежи: "На схеме ЩО-1 указано 3×4"   → partial_consistency.json
3. Критик читает ОБА: "Расхождение текст/чертёж, а не занижение" → review.json
4. Синтезатор: F-001 "Расхождение спецификации и чертежа" → findings.json
```

Правила разрешения конфликтов:
- Два агента нашли одно и то же → дедупликация, берём лучшее описание
- Два агента противоречат → критик разбирает, синтезатор решает
- Критик отверг, но агент прав → синтезатор может восстановить (с обоснованием)
- Нет evidence → автоматическое отклонение

## Архитектурные решения

- **Subagents через файлы**, не Agent Teams — на 25% дешевле при том же качестве
- Раунды 0, 1, 2 максимально параллельны
- Контекст на агента фиксированный (~100K), не растущий
- При падении одного агента — логируем и продолжаем без его результатов
- Система работает полностью автоматически, не задаёт вопросов

## Автономный режим

- Не задаёт вопросов
- Не останавливается на ошибках отдельных агентов
- Все инструменты (Read, Write, Agent) предварительно разрешены
