# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Язык общения:** Всегда отвечай на русском языке.

## Обзор системы

Мультиагентная система аудита проектной документации жилых зданий. Центральный оркестратор запускает параллельных специализированных агентов через Claude Code CLI (`Agent` tool). Каждый агент анализирует проект со своей экспертной позиции, затем агенты-критики перекрёстно проверяют находки, и синтезатор формирует финальный отчёт.

**Принцип:** один MD-файл → N параллельных агентов → спор → синтез → результат.

**Целевой объект:** Жилые многоквартирные дома (МКД), все разделы проектной документации.

## Текущее состояние проекта

**Оркестратор пайплайна — Claude Code (ручной запуск).** Вспомогательные скрипты: gen_report.py (генерация Excel). Ниже — спецификация системы. Существующие артефакты:

## Конвейер аудита

### Раунд 0: Подготовка (оркестратор = Claude Code)

Без отдельного скрипта. Оркестратор (главная сессия Claude Code) сам:
1. Читает `project_info.json` → определяет раздел (section)
2. Читает `document.md` → находит все BLOCK [IMAGE]
3. Запускает **параллельно** N vision-агентов (по одному на каждый BLOCK [IMAGE])
   - Каждый vision-агент читает PNG из `blocks/` и описывает по `prompts/vision_schema.md`
   - Описывает: структуру схем, трассы на планах, компоновку оборудования
   - **НЕ описывает:** штампы, номера листов, таблицы изменений (это уже в document.md)
4. Собирает результаты всех vision-агентов в `_output/structured_blocks.json`
5. По содержимому document.md определяет тип проекта и выбирает какие агенты нужны (не запускает все 14 — только релевантные)
6. Составляет **карту листов** — по заголовкам страниц (`**Лист:** N`, `**Наименование листа:**`) определяет какие листы релевантны какому агенту
7. Запускает раунд 1 — каждому агенту в промпте указывает:
   - `focus_sheets` — листы для подробной проверки
   - `context_sheets` — листы для контекста (общие данные, общие указания)

   Пример: агенту cables → "Фокусируйся на листах 3 (однолинейная схема), проверь по листу 4 (таблица нагрузок). Листы 1-2 (общие данные) — для контекста."

### Раунд 1: Параллельный анализ (до 14 агентов, Sonnet)

Оркестратор выбирает агентов по типу проекта и запускает **параллельно** через Agent tool. Каждому агенту указывает focus_sheets и context_sheets.

Каждый агент получает:
- Путь к `document.md` (читает сам через Read)
- Путь к `_output/structured_blocks.json` (структурированные описания чертежей)
- Свой специализированный промпт из `prompts/`
- Указание на каких листах фокусироваться

Каждый агент **пишет** результат в файл `_output/partial_<name>.json`.

### Раунд 2: Перекрёстная критика (3 критика, Opus)

Три критика запускаются **параллельно**. Каждый проверяет смежные агенты для поиска дублей и противоречий.

```
Критик A                │ Критик B                │ Критик C
Кабели и трассы         │ Расчёты и нормы         │ Спецсистемы
─────────────────────── │ ─────────────────────── │ ───────────────────────
partial_cables.json     │ partial_tables.json     │ partial_lighting.json
partial_fire.json       │ partial_norms.json      │ partial_automation.json
partial_cable_routes.json│ partial_metering.json   │ partial_grounding.json
partial_drawings.json   │ partial_drawings.json   │ partial_outdoor.json
                        │                         │ partial_power_equip.json
Дубли между:            │ Дубли между:            │ Дубли между:
cables ↔ fire (марки)   │ tables ↔ metering (ТТ)  │ lighting ↔ automation
cables ↔ routes (лотки) │ tables ↔ drawings (Pр)  │ outdoor ↔ grounding
fire ↔ routes (EI)      │ norms ↔ все (ссылки)    │ power ↔ lighting (обогр)
                        │                         │
→ review_a.json         │ → review_b.json         │ → review_c.json
```

`drawings` проверяется критиками A и B — он находит расхождения по данным обеих групп.

Вердикты критика:
- `pass` — замечание обосновано и по сути, и по ссылкам
- `pass_weak_norm` — техническая проблема реальна, но norm_ref слабая (идёт в отчёт, синтезатор корректирует ссылку)
- `no_evidence` — нет привязки к документу
- `phantom_block` — block_id не существует в structured_blocks.json
- `weak_evidence` — факты не подтверждают вывод
- `contradicts_drawing` — чертёж/схема опровергает замечание
- `duplicate` — то же самое нашёл другой агент
- `wrong_norm_and_substance` — и норма неверна, и суть замечания ошибочна
- `arithmetic_error` — ошибка в расчёте агента (не проекта)

### Раунд 3: Синтез (1 агент, Opus)

Один агент с высоким качеством рассуждений:

Перед синтезом оркестратор выполняет **предварительную фильтрацию:**
1. Читает все `review_*.json`
2. Удаляет из `partial_*.json` замечания с вердиктом отклонения (no_evidence, phantom_block, weak_evidence, contradicts_drawing, wrong_norm_and_substance, arithmetic_error)
3. Помечает `duplicate` — оставляет оригинал, удаляет дубль
4. Записывает `_output/filtered_findings.json` — только pass + pass_weak_norm
5. Записывает `_output/rejected.json` — все отклонённые с причинами

Синтезатор получает уже отфильтрованные данные:

```
Вход:
  - _output/filtered_findings.json (только принятые замечания)
  - _output/rejected.json (для статистики)
  - document.md (для финальной сверки при сомнениях)

Задачи:
  1. Дедупликация: объединить одинаковые от разных агентов
  2. pass_weak_norm → скорректировать norm_ref
  3. Нумерация: F-001, F-002, ...
  4. Категоризация и приоритизация по severity + confidence
  5. Запись findings.json (включая rejected[] и stats{})

Выход: findings.json — финальный файл замечаний
```

### Раунд 4: Оптимизация (после findings.json)

Оптимизатор ищет возможности экономии, НЕ затрагивая позиции из findings:

```
Вход:
  - findings.json (чтобы не оптимизировать ошибочные позиции)
  - document.md + structured_blocks.json
  - norms/vendor_list.json (допустимые производители)

Задачи:
  1. Замена оборудования на preferred из вендор-листа
  2. Оптимизация кабельных решений (завышенные сечения)
  3. Упрощение конструктивных решений
  4. Привязка к findings категории "Экономическое"

Выход: optimization.json → проверяется критиком → optimization_review.json
```

### Раунд 5: Генерация отчёта (автоматический)

Оркестратор запускает `Audits/gen_report.py` через Bash:

```bash
python3 Audits/gen_report.py
```

Скрипт читает `findings.json` + `optimization.json` + `optimization_review.json` и генерирует Excel-отчёт в формате:
- Лист "СВОДКА" — таблица с эмодзи-заголовками по категориям
- Лист "{project_id}" — все замечания с цветовой маркировкой
- Лист "ОПТ {project_id}" — предложения оптимизации с вердиктами

Выход: `_output/audit_report_{date}.xlsx`

## Агенты по дисциплинам

Оркестратор (Claude Code) решает какие агенты запускать, читая `project_info.json` → `section`. Промпты хранятся в `prompts/`. Сейчас реализованы агенты для раздела EM (электроснабжение):

**Раунд 0:** vision-агенты (параллельно, по одному на BLOCK [IMAGE]) → `structured_blocks.json`
**Раунд 1:** до 14 агентов параллельно (оркестратор выбирает по типу проекта):
- Базовые (все проекты): cables, fire_safety, drawings, tables, norms
- Освещение: lighting
- Наружка: outdoor_install
- Автоматика: automation
- Заземление: grounding
- Силовое: power_equipment
- Учёт: metering
- Трассы: cable_routes
**Раунд 2:** 3 критика параллельно (все Opus):
- critic_a: cables + fire_safety + cable_routes + drawings
- critic_b: tables + norms + metering + drawings
- critic_c: lighting + automation + grounding + outdoor_install + power_equipment + drawings
**Раунд 3:** synthesizer → `findings.json`
**Раунд 4:** optimizer (читает findings + vendor_list) → `optimization.json`
**Раунд 4б:** optimizer_critic → `optimization_review.json`

## Принципы работы агентов

### Единые источники данных

| Тип данных | Авторитетный источник | Кто использует |
|-----------|----------------------|---------------|
| Структура схем (автомат → кабель → потребитель), трассы на планах, компоновка | `_output/structured_blocks.json` | Все агенты раунда 1, критики |
| Текст документа, привязка блоков к листам, штампы, наименования листов, таблицы изменений | `document.md` | Все агенты, критики, синтезатор |
| Статус и редакция нормативных документов | `norms/norms_db.json` | norms, все агенты при ссылке на нормы |
| Проверенные цитаты пунктов норм | `norms/norms_paragraphs.json` | norms, все агенты |
| Допустимые производители оборудования | `norms/vendor_list.json` | optimizer, optimizer_critic |

**Правило:** статус нормы считается непроверенным, пока не подтверждён через `norms_db.json`. Нельзя опираться только на память модели — при отсутствии подтверждения ставь `norm_confidence: 0.5`.

### Главный принцип: аудитор, а не судья

Агенты выявляют потенциальные проблемы и указывают степень уверенности (`confidence`), а не выносят окончательные вердикты. Причины:
- Проектировщик мог применить допустимую методику, не видимую в документе
- Ориентировочные таблицы и коэффициенты в промптах — не абсолютные значения
- "Критическое" — только при явном, неоспоримом несоответствии

### Общие правила (из `prompts/_base.md`)

1. Читай `document.md` + `_output/structured_blocks.json` — основные источники
2. Каждое замечание ОБЯЗАНО иметь `evidence[]` с привязкой к page/block_id
3. Каждое замечание ОБЯЗАНО иметь `confidence` (0.0-1.0)
4. Если не уверен в норме — ставь `norm_confidence < 0.8`
5. НЕ дублируй чужую работу — ищи только то, что указано в твоём промпте
6. Пиши результат в `_output/partial_<name>.json` с обязательным `checklist`
7. Максимум 20 замечаний от одного агента (приоритизируй)

### Для критиков (из `prompts/critic.md`)

1. Читай partial_*.json + document.md + structured_blocks.json для сверки
2. Для каждого замечания — вердикт с обоснованием
3. Если техническая суть верна, но норма слабая → `pass_weak_norm` (НЕ отклоняй)
4. Если находка обосновано — `pass`, не ищи проблемы искусственно

### Для синтезатора (из `prompts/synthesizer.md`)

1. Читай ВСЕ partial_*.json и review_*.json
2. При конфликте критиков — проверь evidence сам и прими решение
3. `pass_weak_norm` → принять, скорректировать norm_ref
4. "Критическое" с confidence < 0.7 → понизить категорию

## Оптимизация

- Агент получает короткий промпт (2-4K) вместо монолитного task-файла
- Агент читает только то, что ему нужно (не весь pipeline output)
- Критик читает partial-файлы, а не полный MD
- Раунды 0, 1, 2 максимально параллельны

## Запуск

Оркестратор — сам Claude Code. Запуск аудита: пользователь указывает путь к проекту, Claude Code читает `project_info.json` и `document.md`, запускает агентов через Agent tool.

Системные требования: Claude Code CLI (установлен глобально).

## Нормативная база

### Правила (наследуются из основного проекта)

1. НЕ ссылайся на устаревшие нормы без пометки о статусе
2. НЕ придумывай номера пунктов — ставь `norm_confidence: 0.5` если не уверен
3. Приоритет: ФЗ → Техрегламенты → СП обязательные → СП добровольные → ГОСТ → ПУЭ
4. При ссылке на ПУЭ-7 для критических решений — подкрепляй параллельной ссылкой на действующий СП/ГОСТ
5. Перед ссылкой — сверяйся с `norms_db.json`

### Файлы

- `norms/norms_db.json` — статус документов (действует/заменён/отменён)
- `norms/norms_paragraphs.json` — проверенные цитаты конкретных пунктов

## Формат замечания

### Категории

| Категория | Описание | Приоритет |
|-----------|----------|-----------|
| Критическое | Явное нарушение норм, влияющее на безопасность | 1 (высший) |
| Экономическое | Влияет на стоимость: пересортица, ошибки при закупке/монтаже | 2 |
| Эксплуатационное | Влияет на работу объекта: проблемы при эксплуатации/безопасности | 3 |

**"Рекомендательное" убрано.** Замечание создаётся только если влияет на стоимость, безопасность или эксплуатацию.

### Обязательные поля

```
id              — F-NNN (назначает синтезатор)
category        — одна из четырёх
title           — краткое название (до 100 символов)
description     — суть замечания
page            — страница PDF (число)
sheet           — лист из штампа (число, может отличаться от page)
evidence[]      — привязка к источнику (type: text|image, block_id, source)
norm_ref        — ссылка на норму (или null)
norm_quote      — цитата пункта (или null)
norm_confidence — 0.0-1.0 (уверенность в правильности нормы)
confidence      — 0.0-1.0 (общая уверенность в замечании)
recommendation  — что делать
source_agents[] — какие агенты нашли (назначает синтезатор)
critic_verdict  — pass|pass_weak_norm|rejected (назначает синтезатор)
```

## Механизм спора (debate protocol)

### Как агенты "спорят"

Агенты не общаются напрямую. Спор реализован через файлы:

```
1. Агент-кабели пишет: "Сечение 2.5 мм², нужно 4 мм²"
   → partial_cables.json

2. Агент-чертежи пишет: "На схеме ЩО-1 указано 3×4"
   → partial_drawings.json

3. Критик А читает ОБА файла:
   → "cables_001 противоречит drawings_007:
      текст говорит 2.5, чертёж говорит 4.
      Вердикт: замечание УТОЧНИТЬ —
      расхождение текст/чертёж, а не занижение."
   → review_a.json

4. Синтезатор читает всё:
   → F-001: "Расхождение спецификации и чертежа:
      в тексте 2.5 мм², на схеме 4 мм².
      Источник: cables + drawings, подтверждено критиком."
   → findings.json
```

### Правила разрешения конфликтов

```
Два агента нашли одно и то же        → дедупликация, берём лучшее описание
Два агента противоречат друг другу   → критик разбирает, синтезатор решает
Критик отверг, но агент прав         → синтезатор может восстановить (с обоснованием)
Критик пропустил, но замечание слабое → синтезатор может отклонить
Нет evidence                          → автоматическое отклонение
```

## Автономный режим

Система работает полностью автоматически:
- Не задаёт вопросов
- Не останавливается на ошибках отдельных агентов
- Если агент упал — логирует и продолжает без его результатов
- Все инструменты (Read, Write, Agent) предварительно разрешены

## Подход к параллелизму: Subagents

Система использует **Subagents** (Agent tool), а не Agent Teams. Протестированы оба подхода:

| Параметр | Subagents | Agent Teams |
|----------|-----------|-------------|
| Токены | 1018K | 1268K (+25%) |
| Время | ~28 мин | ~30 мин |
| Замечаний | 40 | 44 |
| Реальных уникальных | одинаково | +2 опечатки OCR |
| Координация | через файлы | через SendMessage |

**Выбран Subagents** — на 25% дешевле при одинаковом качестве результата. Agent Teams добавляют overhead на координацию, но для нашей задачи агенты не нуждаются в обсуждении — спор через файлы (partial → review → findings) работает так же эффективно.

## Отличия от монолитного пайплайна

| Параметр | Монолитный | Мультиагентный |
|----------|-----------|----------------|
| CLI вызовов | 4-6 последовательных | 5+2+1+1+1 = 10 (5 раундов) |
| Параллелизм | Только блоки | Раунд 1 и 2 полностью параллельны |
| Время | ~30 мин | ~28 мин |
| Перекрёстная проверка | Critic постфактум | Встроена в архитектуру |
| Устойчивость к ошибкам | Весь этап падает | Один агент падает, остальные работают |
| Контекст на агента | Растущий (до 300K) | Фиксированный (~100K) |
| Дедупликация | В findings_merge | В critic + synthesizer |
