# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Язык общения:** Всегда отвечай на русском языке.

## Обзор системы

Мультиагентная система аудита проектной документации жилых зданий (МКД). Центральный оркестратор (Claude Code) запускает параллельных специализированных агентов через `Agent` tool. Каждый агент анализирует проект со своей экспертной позиции, затем агенты-критики перекрёстно проверяют находки, и синтезатор формирует финальный отчёт.

**Принцип:** один MD-файл → N параллельных агентов → спор → синтез → результат.

**Каждая дисциплина** в `prompts/{SECTION}/` содержит:
- `OWNERSHIP_MATRIX_{SECTION}.md` — матрица владения (предотвращает дублирование между агентами)
- `R0_vision_{SECTION}.md` — схема описания чертежей для vision-агентов
- `R1_*.md` — специализированные промпты для агентов раунда 1

**Каждый проект** в `projects/{SECTION}/{PROJECT_ID}/` содержит:
- `project_info.json` — метаданные проекта (section, id, описание)
- `*_document.md` — извлечённый текст из PDF
- `blocks/` — извлечённые изображения из PDF
- `_output/` — все результаты аудита (JSON-файлы + Excel)

## Команды

```bash
# Генерация Excel-отчёта (сейчас пути захардкожены в скрипте — нужно менять перед запуском)
python3 "Claude CLI/Audits/gen_report.py"
```

Основной запуск аудита — через Claude Code CLI: пользователь указывает путь к проекту, оркестратор читает `project_info.json` и запускает пайплайн.

### Формат project_info.json

```json
{
  "project_id": "133_23-ГК-ЭМ2",
  "section": "EOM",
  "md_files": ["Часть 1..._document.md", "Часть 2..._document.md"]
}
```

- `section` — **обязан** совпадать с именем папки в `prompts/` (EOM, AI, AR, VK, OV, TX)
- `md_file` (строка) — один документ, обособленный проект
- `md_files` (массив) — многочастный проект, части идут по порядку (Часть 2 — продолжение Части 1)

**Правило для оркестратора:** если в project_info есть `md_files` — конкатенировать все MD-файлы в один поток перед Раундом 0. Нумерация страниц второго файла продолжает первый. Если есть только `md_file` — один документ.

### Раунд 0: Подготовка и Vision

#### Шаг 0а: Чтение входных данных

1. Читает `project_info.json` → определяет `section` и список MD-файлов
2. Если `md_files` (массив) — конкатенирует все части в один поток (Часть 2 продолжает Часть 1)
3. Если `md_file` (строка) — один документ

#### Шаг 0б: Скачивание IMAGE-блоков

Если папка `_output/blocks/` пуста или отсутствует, оркестратор скачивает кропы изображений:

1. Находит HTML-файл в папке проекта (`*_ocr.html`)
2. Парсит все ссылки вида:
   ```
   href="https://pub-...r2.dev/tree_docs/{DOC_UUID}/crops/{BLOCK_ID}.pdf"
   ```
3. Скачивает каждый PDF-кроп в `_output/blocks/{BLOCK_ID}.pdf`
4. Проверяет: количество скачанных файлов = количество `BLOCK [IMAGE]` в document.md

**Если `_output/blocks/` уже содержит файлы** — пропускает скачивание (блоки уже готовы).

#### Шаг 0в: Vision (Gemini)

На данный момент Gemini-агенты запускаются **отдельно** от оркестратора. В будущем — интеграция в пайплайн.

1. Каждый IMAGE-блок получает свой vision-агент (Gemini)
2. Агент получает кроп из `_output/blocks/{BLOCK_ID}.pdf` + промпт `R0_vision_{SECTION}.md`
3. Результаты собираются в `_output/structured_blocks.json`

**Если `_output/structured_blocks.json` уже существует** — оркестратор пропускает Vision и переходит к шагу 0г.

#### Шаг 0г: Обогащение document.md

Оркестратор создаёт `_output/document_enriched.md` — копию document.md, в которой каждый блок `### BLOCK [IMAGE]: {ID}` заменён на структурированное описание из `structured_blocks.json`.

**Было** (OCR-описание, нечитаемое для анализа):
```markdown
### BLOCK [IMAGE]: 4W69-GN7E-MQ6
**[ИЗОБРАЖЕНИЕ]** | Тип: Схема
**Краткое описание:** Схема расчетная однолинейная ВРУ-НС...
**Описание:** Чертеж представляет собой развернутую однолинейную...
**Сущности:** ВРУ-НС, НС.РП1, НС.РП2...
```

**Стало** (структурированные данные от Vision):
```markdown
### BLOCK [IMAGE]: 4W69-GN7E-MQ6
**Тип:** schema | **Страница:** 10

ИСТОЧНИК:
- Ввод питания от ГРЩ, кабель М-1.6 ППГнг(А)-HF 5х4 L=15м.

ВВОД:
- Щит: ЩСН ЩРН-18-IP31 350x300x120.
- Вводной аппарат: выключатель нагрузки BP, C 40 A.

ОТХОДЯЩИЕ ЛИНИИ:
Линия 1 (Гр.1):
  - Автомат: QF1, BA-103 1P, C 10 A
  - Кабель: ППГнг(А)-HF 3x1.5
  ...
```

**Правила замены:**
1. Найти каждый `### BLOCK [IMAGE]: {ID}` в document.md
2. Найти этот ID в structured_blocks.json
3. Заменить всё содержимое блока (от `### BLOCK [IMAGE]` до следующего `### BLOCK` или `## СТРАНИЦА`) на описание из structured_blocks
4. Если ID не найден в structured_blocks.json → оставить оригинальный OCR-текст, добавить пометку `⚠️ Vision-описание отсутствует`
5. Оригинальный document.md **не изменяется** — enriched создаётся отдельно

**Если `_output/document_enriched.md` уже существует** — пропускает обогащение.

Все последующие шаги (парсинг листов, нарезка, передача агентам) работают с `document_enriched.md`, а не с оригинальным `document.md`.

#### Шаг 0д: Парсинг листов

Оркестратор читает `document_enriched.md` и строит индекс листов. Маркер листа:

```
## СТРАНИЦА {N}
**Лист:** {номер}
**Наименование листа:** {название}
```

Для каждого листа фиксируется: номер страницы, номер листа, наименование, диапазон строк (от `## СТРАНИЦА N` до следующего `## СТРАНИЦА`).

#### Шаг 0е: Классификация листов

#### Шаг 0ж: Маппинг категорий → агентов

Для каждого R1-агента определено, какие категории листов ему нужны. **Общие листы** (`general_notes`, `specification`) дублируются всем агентам — это ожидаемо и правильно.

**EOM (электроснабжение):**

| Агент | required | optional |
|-------|----------|----------|
| `cables` | `schema`, `specification`, `general_notes` | `cable_journal`, `load_table` |
| `fire_safety` | `schema`, `specification`, `general_notes`, `fire_safety` | `lighting_plan`, `cable_route_plan` |
| `drawings` | `toc`, `schema`, `specification`, `general_notes` | все остальные (получает ВЕСЬ document.md) |
| `tables` | `load_table`, `schema`, `specification`, `general_notes` | `cable_journal` |
| `norms` | все (получает ВЕСЬ document.md) | — |
| `lighting` | `lighting_plan`, `schema`, `specification`, `general_notes` | `facade_plan` |
| `cable_routes` | `cable_route_plan`, `specification`, `general_notes` | `power_plan`, `lighting_plan` |
| `automation` | `schema`, `specification`, `general_notes`, `automation` | `lighting_plan` |
| `grounding` | `grounding_plan`, `schema`, `specification`, `general_notes` | `section_drawing` |
| `power_equipment` | `schema`, `power_plan`, `specification`, `general_notes` | `load_table` |
| `metering` | `schema`, `metering`, `specification`, `general_notes` | `load_table` |
| `outdoor_install` | `outdoor`, `specification`, `general_notes` | `cable_route_plan` |

Для других дисциплин (AI, AR, VK, OV, TX) маппинг аналогичен и определяется в `prompts/{SECTION}/SHEET_MAP_{SECTION}.md` (если создан) или оркестратор строит его по аналогии.

**Агенты-исключения** (получают ВЕСЬ document.md):
- `norms` — ищет ссылки на нормы по всему документу
- `drawings` — сверяет ведомость чертежей с реальными листами

Для крупных документов (>500 KB) эти агенты могут запускаться на модели с большим контекстом.

#### Шаг 0з: Сборка среза и формирование промпта

Для каждого R1-агента оркестратор:

1. Берёт листы из `required` категорий → основной контент
2. Если контекст позволяет (<150K токенов с учётом промпта и norms_db) → добавляет `optional`
3. Собирает срез document.md (конкатенация выбранных листов с сохранением оригинальной нумерации строк и маркеров `## СТРАНИЦА`)
4. Формирует промпт агента в порядке:

```
[norms_db.json]                              ← общий префикс (кэшируется)
[norms_paragraphs.json]                      ← общий префикс (кэшируется)
[срез document_enriched.md для этого агента]  ← содержит текст + описания чертежей
[00_base.md]                                 ← общий (кэшируется)
[OWNERSHIP_MATRIX_{SECTION}.md]              ← общий для секции (кэшируется)
[R1_{agent}.md]                              ← уникальный
```

**Порядок важен для prompt caching:** общие блоки идут первыми, уникальные — в конце.

#### Шаг 0и: Проверка покрытия (серые зоны)

После маппинга оркестратор проверяет:

```
Для каждого листа в индексе:
  если лист (по его категориям) не попадает ни в один required/optional ни одного агента:
    → WARNING: "Лист {N} '{наименование}' — серая зона, не покрыт ни одним агентом"
    → Оркестратор назначает лист ближайшему по тематике агенту в его optional
```

Ни один лист не должен остаться без проверки. `uncategorized` листы назначаются агенту `drawings` (он проверяет комплектность).

### Раунд 1: Анализ (до 14 агентов, параллельно)

Все R1-агенты запускаются **одновременно** — ограничений на количество параллельных subagents нет. Агенты полностью независимы друг от друга, зависимости между ними отсутствуют.

**Выбор модели:** по умолчанию Sonnet. Если общий размер промпта агента (срез document.md + structured_blocks.json + norms_db.json + промпт) превышает **200K токенов** → агент запускается на **Opus** (контекст 1M).

Каждый агент получает **в промпте** (не читает сам через Read):
- Срез `document_enriched.md` (подготовлен на шаге 0з) — уже содержит описания чертежей из structured_blocks
- `norms/norms_db.json` и `norms/norms_paragraphs.json`
- Свой промпт из `prompts/{SECTION}/R1_*.md`

Каждый агент **пишет** результат в `_output/partial_<name>.json`.

#### Обратная связь: missing_data

Если агент не нашёл в предоставленном срезе данные, необходимые для выполнения шага, он сообщает об этом в чеклисте:

```json
"checklist": {
  "step_1_data_collected": {
    "done": true,
    "notes": "...",
    "missing_data": [
      {
        "description": "Кабельный журнал не найден в предоставленных листах",
        "expected_content": "Таблица с перечнем кабелей, длинами и способами прокладки",
        "impact": "Невозможно проверить длины кабелей",
        "steps_affected": ["step_5_installation"]
      }
    ]
  }
}
```

**Логика перезапуска оркестратором:**

1. После завершения всех R1-агентов оркестратор проверяет `missing_data` в каждом `partial_*.json`
2. Если агент запросил данные, и соответствующий лист **существует** в индексе, но не был передан → оркестратор **перезапускает** агента с расширенным срезом
3. Если лист **не существует** в документе → это валидная информация (неполнота комплекта), агент фиксирует это в notes
4. **Максимум 1 перезапуск** на агента — если после второй попытки снова missing_data, оркестратор логирует и продолжает

### Раунд 2: Критика (3 критика Opus, параллельно)

Три критика проверяют смежные агенты для поиска дублей и противоречий. `drawings` проверяется критиками A и B — он находит расхождения по данным обеих групп.

Вердикты критика:
- `pass` — замечание обосновано и по сути, и по ссылкам
- `pass_weak_norm` — техническая проблема реальна, но norm_ref слабая
- `no_evidence` — нет привязки к документу
- `phantom_block` — block_id не существует в structured_blocks.json
- `weak_evidence` — факты не подтверждают вывод
- `contradicts_drawing` — чертёж опровергает замечание
- `duplicate` — то же самое нашёл другой агент
- `wrong_norm_and_substance` — и норма неверна, и суть ошибочна
- `arithmetic_error` — ошибка в расчёте агента (не проекта)

### Раунд 3: Синтез (1 агент Opus)

Перед синтезом оркестратор **фильтрует:**
1. Удаляет замечания с вердиктом отклонения → `_output/rejected.json`
2. Оставляет pass + pass_weak_norm → `_output/filtered_findings.json`

Синтезатор: дедупликация → коррекция norm_ref → нумерация F-001..F-NNN → приоритизация → `findings.json`

### Раунд 4: Оптимизация

Оптимизатор ищет экономию, НЕ затрагивая позиции из findings:
- Замена оборудования на preferred из вендор-листа
- Оптимизация завышенных сечений
- Упрощение конструктивных решений

→ `optimization.json` → критик → `optimization_review.json`

### Раунд 5: Отчёт

`gen_report.py` читает findings + optimization + optimization_review → Excel с листами "СВОДКА", "{project_id}", "ОПТ {project_id}".

## Принципы работы агентов

### Единые источники данных

| Тип данных | Источник | Кто использует |
|-----------|---------|---------------|
| Текст + структура чертежей (единый файл) | `_output/document_enriched.md` | Все агенты R1 (в виде среза), критики |
| Структура схем (оригинал, до обогащения) | `_output/structured_blocks.json` | Оркестратор (для шага 0г), критики (для верификации block_id) |
| Статус нормативных документов | `norms/norms_db.json` | norms, все агенты при ссылке на нормы |
| Проверенные цитаты пунктов | `norms/norms_paragraphs.json` | norms, все агенты |
| Допустимые производители | `norms/vendor_list.json` | optimizer, optimizer_critic |

### Главный принцип: аудитор, а не судья

Агенты выявляют потенциальные проблемы и указывают степень уверенности (`confidence`), а не выносят окончательные вердикты. "Критическое" — только при явном, неоспоримом несоответствии.

### Общие правила (из `prompts/common/00_base.md`)

1. Читай `document.md` + `structured_blocks.json` — основные источники
2. Каждое замечание ОБЯЗАНО иметь `evidence[]` с привязкой к page/block_id
3. Каждое замечание ОБЯЗАНО иметь `confidence` (0.0-1.0)
4. Если не уверен в норме — ставь `norm_confidence < 0.8`
5. НЕ дублируй чужую работу — ищи только то, что указано в твоём промпте
6. Пиши результат в `_output/partial_<name>.json` с обязательным `checklist`
7. Максимум 20 замечаний от одного агента (приоритизируй)

### Категории замечаний

| Категория | Описание | Приоритет |
|-----------|----------|-----------|
| Критическое | Явное нарушение норм, влияющее на безопасность | 1 (высший) |
| Экономическое | Влияет на стоимость: пересортица, ошибки при закупке/монтаже | 2 |
| Эксплуатационное | Влияет на работу объекта: проблемы при эксплуатации | 3 |

### Нормативная база

1. НЕ ссылайся на устаревшие нормы без пометки о статусе
2. НЕ придумывай номера пунктов — ставь `norm_confidence: 0.5` если не уверен
3. Приоритет: ФЗ → Техрегламенты → СП обязательные → СП добровольные → ГОСТ → ПУЭ
4. При ссылке на ПУЭ-7 — подкрепляй параллельной ссылкой на действующий СП/ГОСТ
5. Перед ссылкой — сверяйся с `norms_db.json`

## Механизм спора (debate protocol)

Агенты не общаются напрямую. Спор реализован через файлы:

```
1. Агент-кабели: "Сечение 2.5 мм², нужно 4 мм²" → partial_cables.json
2. Агент-чертежи: "На схеме ЩО-1 указано 3×4"   → partial_drawings.json
3. Критик А читает ОБА: "Расхождение текст/чертёж, а не занижение" → review_a.json
4. Синтезатор: F-001 "Расхождение спецификации и чертежа" → findings.json
```

Правила разрешения конфликтов:
- Два агента нашли одно и то же → дедупликация, берём лучшее описание
- Два агента противоречат → критик разбирает, синтезатор решает
- Критик отверг, но агент прав → синтезатор может восстановить (с обоснованием)
- Нет evidence → автоматическое отклонение

## Архитектурные решения

- **Subagents через файлы**, не Agent Teams — на 25% дешевле при том же качестве
- Раунды 0, 1, 2 максимально параллельны
- Контекст на агента фиксированный (~100K), не растущий
- При падении одного агента — логируем и продолжаем без его результатов
- Система работает полностью автоматически, не задаёт вопросов

## Автономный режим

- Не задаёт вопросов
- Не останавливается на ошибках отдельных агентов
- Все инструменты (Read, Write, Agent) предварительно разрешены
